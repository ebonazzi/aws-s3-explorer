# Cross-Platform Application Icons Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Embed the existing `assets/` icon files so the app shows a window/taskbar icon on all platforms, a Windows `.exe` icon, a macOS `.app` bundle icon, and a Linux desktop-launcher icon.

**Architecture:** Four independent, additive changes, one per concern: (1) `src/main.rs` sets a runtime `egui::IconData` via `eframe::icon_data::from_png_bytes` on `ViewportBuilder`; (2) a new root `build.rs` uses `winresource` (Windows-only build-dependency, gated by `cfg(windows)`) to embed `assets/icon.ico` into the `.exe`; (3) a new `[package.metadata.bundle]` Cargo.toml section points the external `cargo-bundle` tool at `assets/icon.icns`; (4) new `packaging/linux/` files (`.desktop.in` template + `install.sh`) generate a `~/.local/share/applications/aws-s3-explorer.desktop` entry with an absolute path to `assets/icon.png`. None of the four changes depend on each other — any can be built and tested in isolation.

**Tech Stack:** `eframe`/`egui` 0.35.0 (already a dependency — `icon_data::from_png_bytes` requires no new runtime crate), `winresource = "0.1"` (new Windows-only build-dependency), `cargo-bundle` (external dev-time CLI tool, not a Cargo.toml dependency), bash + `sed` (Linux packaging script).

## Global Constraints

- Runtime icon is set once at startup via `ViewportBuilder::with_icon` — no runtime icon swapping (egui's `ViewportCommand::Icon` has known reliability issues on macOS/Windows and this app never needs to change its icon after launch).
- `winresource` must be a **Windows-only** build-dependency via `[target.'cfg(windows)'.build-dependencies]` — it must not compile or even resolve as a dependency on Linux/macOS targets.
- Use the existing `assets/icon.ico` as-is via `set_icon(...)` — do not regenerate or re-derive it; it already contains multiple embedded resolutions (16×16, 32×32 confirmed).
- `cargo-bundle`'s `icon` field points directly at `assets/icon.icns` (not a list of PNGs) — this uses the hand-produced `.icns` as-is rather than having `cargo-bundle` regenerate one from PNGs.
- Bundle `identifier = "com.ebonazzi.aws-s3-explorer"` — a placeholder reverse-DNS bundle ID, trivially changeable later, with no functional effect beyond macOS's own app-identity bookkeeping.
- Linux desktop icon uses an **absolute path** to `assets/icon.png` in the `.desktop` file's `Icon=` field — no XDG hicolor theme installation, no icon-cache refresh step.
- No new runtime dependency is needed for the runtime icon — `eframe` already vendors `image` for exactly this purpose.
- This is a Linux dev machine: only the runtime icon (Task 1) and the Linux desktop file (Task 4) can be end-to-end verified here. Tasks 2 and 3 are verified for correctness of the Linux-visible parts (compiles, dependency graph is correctly gated, TOML parses) — the actual Windows `.exe` and macOS `.app` can only be produced and confirmed on those OSes.

- [ ] **Step 2: Write `build.rs`**

Create `build.rs` at the project root (same directory as `Cargo.toml`):

```rust
fn main() {
    if std::env::var("CARGO_CFG_TARGET_OS").as_deref() == Ok("windows") {
        winresource::WindowsResource::new()
            .set_icon("assets/icon.ico")
            .compile()
            .expect("failed to embed Windows icon resource");
    }
}
```

Cargo automatically detects and runs a `build.rs` at the project root — no `build = "build.rs"` field is needed in `Cargo.toml`'s `[package]` section.

### Task 4: Linux desktop launcher icon

**Files:**
- Create: `packaging/linux/aws-s3-explorer.desktop.in`
- Create: `packaging/linux/install.sh`

**Interfaces:**
- Consumes: `assets/icon.png` (same file as Task 1) and the release binary at `target/release/aws-s3-explorer` (built by `cargo build --release`, already a project convention per `src/main.rs`'s header comment).
- Produces: nothing consumed by later tasks — self-contained. This is the last task in the plan.

- [ ] **Step 1: Create the `.desktop` template**

Create `packaging/linux/aws-s3-explorer.desktop.in`:

```ini
[Desktop Entry]
Type=Application
Name=AWS S3 Explorer
Comment=Personal AWS S3 / local file manager
Exec=__EXEC_PATH__
Icon=__ICON_PATH__
Terminal=false
Categories=Utility;FileTools;
```

- [ ] **Step 2: Create the install script**

Create `packaging/linux/install.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail

repo_root="$(cd "$(dirname "${BASH_SOURCE[0]}")/../.." && pwd)"
exec_path="$repo_root/target/release/aws-s3-explorer"
icon_path="$repo_root/assets/icon.png"

if [ ! -x "$exec_path" ]; then
    echo "error: $exec_path not found — run 'cargo build --release' first" >&2
    exit 1
fi

dest_dir="$HOME/.local/share/applications"
mkdir -p "$dest_dir"

sed \
    -e "s|__EXEC_PATH__|$exec_path|" \
    -e "s|__ICON_PATH__|$icon_path|" \
    "$repo_root/packaging/linux/aws-s3-explorer.desktop.in" \
    > "$dest_dir/aws-s3-explorer.desktop"

echo "Installed $dest_dir/aws-s3-explorer.desktop"
```

- [ ] **Step 3: Make the install script executable**

Run: `chmod +x packaging/linux/install.sh`

- [ ] **Step 4: Ensure a release binary exists, then run the install script**

Run: `cargo build --release`
Run: `./packaging/linux/install.sh`
Expected output: `Installed /home/<user>/.local/share/applications/aws-s3-explorer.desktop` (no errors).

- [ ] **Step 5: Verify the generated desktop file's contents**
